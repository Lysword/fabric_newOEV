```markdown
## 一、当前系统背景

当前系统整体流程大致为：

1. 背书阶段先执行智能合约；
2. 执行过程中产生读写集；
3. 交易进入排序阶段；
4. 排序后进入验证与提交阶段；
5. 验证阶段基于读写集进行冲突检测；
6. 提交阶段将写集落库。

## 二、目标架构调整方向

我希望将当前“背书阶段真实执行合约并产生读写集”的模式，改造成类似：

> 背书阶段提取读写意图和执行计划，排序阶段基于超立方体进行冲突检测和批次排序，验证/提交阶段根据计划进行重放或落库。

也就是：

### 1. 背书阶段：读写意图提取

在背书阶段，不再以普通方式直接和底层状态数据库进行完整业务交互，而是：

- 基于智能合约执行流程，解析/预测该交易可能读取的 key；
- 解析/预测该交易可能写入的 key；
- 提取写入 value 或者提取可重放的操作逻辑；
- 暂存交易的执行计划；
- 暂存读 key、写 key、操作类型、操作参数等信息；
- 将这些信息作为后续构建读写超立方体的依据；
- 用于排序阶段分析同一 block 内交易之间是否可能冲突。

这里可以采用“读写意图重放”方案：

- 背书阶段提取预测读键、写键和操作值/操作逻辑；
- 提交阶段不再真正重新执行业务合约；
- 提交阶段只根据背书阶段提取出的计划或写集落库；
- 如果 value 依赖读取值，可以在验证/提交阶段根据计划重放计算；
- 但为快速实现，也可以针对 `smallbank`、`ycsb` 的操作特点做特化逻辑。

需要重点考虑：

- 是否只对特定合约启用新逻辑；
- 如何避免影响系统链码；
- 如何在普通链码和 benchmark 链码之间做分流；
- 哪些合约使用新路径，哪些合约仍走原 Fabric 原生流程。

## 三、排序阶段：基于超立方体的冲突检测与批次排序

背书阶段提取到每笔交易的读写 key / 超立方体信息后，在 orderer 排序阶段进行冲突检测和冲突消解。

排序阶段逻辑要求：

1. 分析范围仅限同一个 block 内的交易；
2. 不考虑跨 block 的超立方体冲突；
3. 根据每笔交易的读写超立方体判断交易之间是否相交；
4. 如果两个交易的超立方体相交，则认为它们可能冲突；
5. 基于交易冲突关系构建无向图：
   - 节点：交易；
   - 边：两个交易之间可能冲突；
6. 将冲突消解建模为图染色问题：
   - 颜色表示批次编号；
   - 同一颜色/同一批次内的交易应尽量无冲突；
   - 不同颜色/不同批次之间认为可能存在冲突；
   - 目标是尽量减少颜色数量，即尽量减少批次数；
7. orderer 需要输出一个明确的多批次交易执行顺序，例如：

   batch 0: tx1, tx5, tx8
   batch 1: tx2, tx3
   batch 2: tx4, tx6, tx7
```

1. **同批次交易可以在验证/提交阶段并行处理；**
2. **不同批次必须严格按照 orderer 生成的顺序串行推进；**
3. **后一个批次应视为在前一个批次提交结果基础上继续执行；**
4. **排序结果必须明确传递给 peer 侧的验证/提交阶段。**

**请重点分析：**

- **orderer 当前在哪里组 block；**
- **如何在 block 中附加批次排序结果；**
- **是否需要修改 block metadata；**
- **是否需要新增 protobuf 字段；**
- **或者是否可以用 transaction/action 的 transient/extension 字段、metadata 字段或自定义 envelope 扩展来快速实现；**
- **哪种实现最小侵入、最适合快速落地。**

## **四、验证与提交阶段：按排序批次重放执行计划**

**验证/提交阶段需要根据 orderer 给出的批次排序结果进行处理：**

1. **获取当前 block 内的交易批次划分；**
2. **按照 batch 顺序依次处理；**
3. **同一个 batch 内的交易可以并行处理；**
4. **batch 之间必须串行；**
5. **对启用新架构的 benchmark 合约：**

- **不再完整调用原始智能合约业务逻辑；**
- **读取背书阶段提取出的执行计划、读写 key、操作参数；**
- **根据计划进行重放；**
- **或者直接根据提取出的写集落库；**

1. **对系统链码和非目标合约：**

- **保持原有 Fabric 验证与提交路径；**
- **不应破坏原系统功能。**

**需要明确：**
